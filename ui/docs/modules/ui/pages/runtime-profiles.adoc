= Runtime Profiles
:description: UI runtime profile responsibilities.

Runtime profiles describe the services the UI should use.
Managed services include a working directory, package path, and argument list.
External services set auto-start to false and provide only connection details.
Gateway profiles also name the beta status endpoint and the non-secret model/provider identifier shown to operators.

== Shipped Profiles

[cols="1,2",options="header"]
|===
|Profile |Use

|`runtime_profiles/agent_awesome.json`
|Default local Agent Awesome topology.

|`runtime_profiles/local_dev.json`
|Local memory with the harness pointed at `harness/tool.local.yaml`.

|`runtime_profiles/cloudflare_context.json`
|External Cloudflare gateway used by the desktop UI without starting local services.

|===

Use profiles to change service topology without mixing deployment policy into widgets.
Each profile owns a `memory_domains` list and an `agent_memory` grant block.
Memory domains are user-defined profile data; domain labels such as family, work, or project are not encoded in application logic.
The default shipped profile defines one domain named `memory`.

The default profile reads model, agent, and tool config from the harness package.
Keep `harness/model.yaml`, `harness/agent.yaml`, and `harness/tool.local.yaml` present when packaging or distributing the desktop app.
The gateway status URL resolves to `/api/gateway/beta-status`, which returns authenticated health, Slack scope, snapshot freshness, and model identity without exposing secrets.

== Cloud Gateway Profile

Use `runtime_profiles/cloudflare_context.json` when the desktop UI should talk to a hosted Cloudflare deployment.
Set `AGENT_GATEWAY_BASE_URL` to the hosted `/api` route, set `AGENTAWESOME_GATEWAY_TOKEN` to the Worker gateway bearer token, and set `AGENTAWESOME_RUNTIME_PROFILE` to the cloud profile path.
The cloud profile keeps `auto_start` disabled for the harness, gateway, and memory domain because those services are owned by the Cloudflare Container.
