= Deploy Cloudflare Pilot
:description: Operations deployment commands for the Cloudflare pilot.

Run deployment commands from `deploy/cloudflare/worker`.
Use Node.js 22 or newer for Wrangler.
Make sure Docker is running before `wrangler deploy`; Wrangler builds and pushes the configured Container image during deployment.

The beta container defaults to OpenAI `gpt-5.4-mini`. If you edit
`AGENTAWESOME_MODEL_ID` in `deploy/cloudflare/worker/wrangler.jsonc`, verify the
deployed `OPENAI_API_KEY` can access that model before deploying. The container
renders `/app/runtime/model.yaml` from Worker vars at startup and logs the actual
provider/model pair.

== Local Container Check

Run these commands from the repository root before deploying:

[source,sh]
----
docker build --file Dockerfile.cloudflare --tag agent-awesome-cloudflare:debug .
docker run --rm -d \
  --name agent-awesome-cloudflare-debug \
  -p 8070:8070 \
  -e AGENTAWESOME_GATEWAY_TOKEN=test \
  -e AGENTAWESOME_PERSISTENCE_TOKEN=test \
  -e OPENAI_API_KEY="${OPENAI_API_KEY:-test}" \
  -e SLACK_ENABLED=false \
  agent-awesome-cloudflare:debug
curl -fsS http://127.0.0.1:8070/healthz
curl -fsS \
  -H "Authorization: Bearer test" \
  http://127.0.0.1:8070/api/gateway/beta-status
docker stop agent-awesome-cloudflare-debug
----

== Install And Deploy

Generate the two internal bearer tokens before setting Worker secrets:

[source,sh]
----
GATEWAY_TOKEN="$(openssl rand -base64 32 | tr '+/' '-_' | tr -d '=')"
PERSISTENCE_TOKEN="$(openssl rand -base64 32 | tr '+/' '-_' | tr -d '=')"
----

[source,sh]
----
nvm use
npm --prefix deploy/cloudflare/worker install --cache ../../build/npm-cache
npm run cloudflare:test
npm run cloudflare:r2:create
printf '%s\n' "$GATEWAY_TOKEN" | npm --prefix deploy/cloudflare/worker exec wrangler secret put AGENTAWESOME_GATEWAY_TOKEN
printf '%s\n' "$PERSISTENCE_TOKEN" | npm --prefix deploy/cloudflare/worker exec wrangler secret put AGENTAWESOME_PERSISTENCE_TOKEN
npm --prefix deploy/cloudflare/worker exec wrangler secret put OPENAI_API_KEY
npm run cloudflare:deploy
----

Run those commands from the repository root. If your shell is already in `deploy/cloudflare/worker`, use `npm test`, `npm run r2:create`, `npx wrangler secret put ...`, and `npm run deploy` instead. Do not combine `cd deploy/cloudflare/worker` with `--config deploy/cloudflare/worker/wrangler.jsonc`; Wrangler resolves that relative path under the Worker directory.

The package deploy scripts run Wrangler from the Worker directory and keep Wrangler and Docker state under `build/`, so they also work in shells where home-directory config paths are read-only.

If you use an API token instead of `wrangler login`, export these values before the Wrangler commands:

[source,sh]
----
export CLOUDFLARE_ACCOUNT_ID=<account-id>
export CLOUDFLARE_API_TOKEN=<token>
----

After first deployment, enable Slack only when channel routing is configured, then set the Slack Events API request URL to:

[source,text]
----
https://agent-awesome.com/slack/events
----

Slack is disabled by default so the gateway can deploy before channel routing is configured.
For the workspace-scoped beta, upload `SLACK_ENABLED`, `SLACK_SIGNING_SECRET`, and `SLACK_BOT_TOKEN` as Worker secrets.
Signed events from the installed Slack app route to the default `doug` profile.
For multi-profile Slack, set `SLACK_ENABLED` to `true`, edit `AGENTAWESOME_AGENT_PROFILES_JSON` in `wrangler.jsonc`, and add `slack_bindings` to the profile that owns the channel.
Each binding must include `team_id`, `channel_id`, and `allowed_user_ids`.
The older `SLACK_ALLOWED_TEAM_ID`, `SLACK_ALLOWED_USER_ID`, and `SLACK_ALLOWED_CHANNEL_ID` values still optionally scope Slack to the default profile.

The public `workers.dev` URL is disabled.
The configured production route is `agent-awesome.com/*`.
The checked-in Worker config uses a beta sandbox R2 bucket; provisioner-generated deployments use per-agent buckets.

The `npm test` smoke test typechecks and bundles the Worker, verifies the
Container and Durable Object declarations in `wrangler.jsonc`, confirms
`/healthz` reaches the gateway, confirms `/mcp` is forwarded only with
`AGENTAWESOME_GATEWAY_TOKEN`, and confirms signed Slack Events API requests
reach the gateway path.

After deploy, wait several minutes for the first Container rollout, then inspect status:

[source,sh]
----
npx wrangler containers list
npx wrangler containers images list
----

== Verify Gateway Access

Check public liveness, then check the authenticated gateway surfaces used by the desktop UI:

[source,sh]
----
curl -fsS https://agent-awesome.com/healthz
curl -fsS \
  -H "Authorization: Bearer ${GATEWAY_TOKEN}" \
  https://agent-awesome.com/api/gateway/beta-status
curl -fsS \
  -H "Authorization: Bearer ${GATEWAY_TOKEN}" \
  -H "X-Agent-Awesome-Profile: doug" \
  https://agent-awesome.com/api/context/tools/list
----

== Connect Desktop UI

Start the release binary with the Cloudflare runtime profile and the same gateway token:

[source,sh]
----
export AGENT_GATEWAY_BASE_URL=https://agent-awesome.com/api
export AGENTAWESOME_GATEWAY_TOKEN="${GATEWAY_TOKEN}"
export AGENTAWESOME_RUNTIME_PROFILE="<release-root>/ui/runtime_profiles/cloudflare_context.json"
export AUTO_START_LOCAL_SERVICES=false
./agentawesome_ui
----

The desktop UI sends chat, memory, and task traffic to the hosted gateway.
It does not call memory MCP endpoints directly.
Use `ui/runtime_profiles/cloudflare_family.json` to connect the same release binary to the Family profile.

== Verify Slack Channel

Keep the Slack Events API request URL at:

[source,text]
----
https://agent-awesome.com/slack/events
----

Send one message to the installed Slack app, then inspect Worker logs:

[source,sh]
----
npx wrangler tail agent-awesome-pilot --format pretty
----

Slack traffic enters through the Worker signature check and then reaches the same hosted gateway as the desktop UI.
