= Deploy Cloudflare Pilot
:description: Operations deployment commands for the Cloudflare pilot.

Run deployment commands from `deploy/cloudflare/worker`.
Use Node.js 22 or newer for Wrangler.
Make sure Docker is running before `wrangler deploy`; Wrangler builds and pushes the configured Container image during deployment.

== Install And Deploy

Generate the two internal bearer tokens before setting Worker secrets:

[source,sh]
----
GATEWAY_TOKEN="$(openssl rand -base64 32 | tr '+/' '-_' | tr -d '=')"
PERSISTENCE_TOKEN="$(openssl rand -base64 32 | tr '+/' '-_' | tr -d '=')"
----

[source,sh]
----
npm install --cache ../../../build/npm-cache
npm test
npx wrangler r2 bucket create agent-awesome-beta-context
printf '%s\n' "$GATEWAY_TOKEN" | npx wrangler secret put AGENTAWESOME_GATEWAY_TOKEN
printf '%s\n' "$PERSISTENCE_TOKEN" | npx wrangler secret put AGENTAWESOME_PERSISTENCE_TOKEN
npx wrangler secret put SLACK_SIGNING_SECRET
npx wrangler secret put SLACK_BOT_TOKEN
npx wrangler secret put OPENAI_API_KEY
npx wrangler deploy
----

If you use an API token instead of `wrangler login`, export these values before the Wrangler commands:

[source,sh]
----
export CLOUDFLARE_ACCOUNT_ID=<account-id>
export CLOUDFLARE_API_TOKEN=<token>
----

After first deployment, set the Slack Events API request URL to:

[source,text]
----
https://agent-awesome.com/slack/events
----

For multi-profile Slack, edit `AGENTAWESOME_AGENT_PROFILES_JSON` in `wrangler.jsonc` and add `slack_bindings` to the profile that owns the channel.
Each binding must include `team_id`, `channel_id`, and `allowed_user_ids`.
The older `SLACK_ALLOWED_TEAM_ID`, `SLACK_ALLOWED_USER_ID`, and `SLACK_ALLOWED_CHANNEL_ID` values scope Slack to the default profile only.
The gateway refuses to start Slack ingress unless either a profile binding or a complete legacy allow-list is configured.

The public `workers.dev` URL is disabled.
The configured production route is `agent-awesome.com/*`.
The checked-in Worker config uses a beta sandbox R2 bucket; provisioner-generated deployments use per-agent buckets.

The `npm test` smoke test typechecks and bundles the Worker, verifies the
Container and Durable Object declarations in `wrangler.jsonc`, confirms
`/healthz` reaches the gateway, confirms `/mcp` is forwarded only with
`AGENTAWESOME_GATEWAY_TOKEN`, and confirms signed Slack Events API requests
reach the gateway path.

After deploy, wait several minutes for the first Container rollout, then inspect status:

[source,sh]
----
npx wrangler containers list
npx wrangler containers images list
----

== Verify Gateway Access

Check public liveness, then check the authenticated gateway surfaces used by the desktop UI:

[source,sh]
----
curl -fsS https://agent-awesome.com/healthz
curl -fsS \
  -H "Authorization: Bearer ${GATEWAY_TOKEN}" \
  https://agent-awesome.com/api/gateway/beta-status
curl -fsS \
  -H "Authorization: Bearer ${GATEWAY_TOKEN}" \
  -H "X-Agent-Awesome-Profile: doug" \
  https://agent-awesome.com/api/context/tools/list
----

== Connect Desktop UI

Start the release binary with the Cloudflare runtime profile and the same gateway token:

[source,sh]
----
export AGENT_GATEWAY_BASE_URL=https://agent-awesome.com/api
export AGENTAWESOME_GATEWAY_TOKEN="${GATEWAY_TOKEN}"
export AGENTAWESOME_RUNTIME_PROFILE="<release-root>/ui/runtime_profiles/cloudflare_context.json"
export AUTO_START_LOCAL_SERVICES=false
./agentawesome_ui
----

The desktop UI sends chat, memory, and task traffic to the hosted gateway.
It does not call memory MCP endpoints directly.
Use `ui/runtime_profiles/cloudflare_family.json` to connect the same release binary to the Family profile.

== Verify Slack Channel

Keep the Slack Events API request URL at:

[source,text]
----
https://agent-awesome.com/slack/events
----

Send one message from the allow-listed Slack user in the allow-listed channel, then inspect Worker logs:

[source,sh]
----
npx wrangler tail agent-awesome-pilot --format pretty
----

Slack traffic enters through the Worker signature check and then reaches the same hosted gateway as the desktop UI.
