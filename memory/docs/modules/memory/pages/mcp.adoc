= Memory MCP
:description: Memory service MCP responsibilities.

The memory MCP transport adapts service capabilities into tools that the harness can call.
It should validate input at the domain and service layers rather than trusting callers.

== Local Endpoint

Local profiles usually run memory on:

[source,text]
----
http://127.0.0.1:8090/mcp
----

The UI and gateway profiles point to this endpoint when local memory is active.

== Boundary

MCP code belongs in `internal/memory/transport`.
Business behavior belongs in `internal/memory/service`.
DTO validation belongs in `internal/memory/domain` or graph-specific domain packages.

== Small Memory Nuggets

Local micro models should use `remember` when the user asks the assistant to remember a fact, preference, note, or small piece of knowledge.
`remember` intentionally accepts a tiny payload:

* `text`: the memory nugget to preserve.
* `title`: optional display title.
* `topics`: optional topic tags for later connection.
* `entities`: optional people, projects, places, or things mentioned.
* `scope`, `sensitivity`, `idempotency_key`, and `actor`: optional control fields.

The transport maps that small payload into the richer graph-backed memory model as user-asserted private memory by default.
Use `create_task` only for operational todos, not for facts or preferences.

== Memory Maintenance

Use `organize_memory` for bounded maintenance passes over a memory domain.
The tool reviews existing records, fills deterministic source-backed summaries when that is safe, and creates idempotent clarification tasks for records that need a better title, topics, subjects, or additional detail.
Pass `dry_run: true` to preview the questions and safe summary repairs without writing memory or task changes.

== Low-Friction Tasks

Local micro models should use `create_task` when the user asks for an operational todo, even if the user only provides a short phrase such as "buy milk".
The model-facing payload is intentionally small:

* `title`: the task to do.
* `description`: optional note only when the user provides one.
* `priority`: optional `low`, `normal`, `high`, or `urgent`; default is `normal`.
* `due_at`: optional deadline timestamp when the user gives a due date.
* `scheduled_at`: optional start or reminder timestamp when the user gives one.
* `topics`, `idempotency_key`, and `actor`: optional control fields.

The transport only maps those schema fields during creation.
Richer planning fields such as urgency, project, location, memory links, and work-breakdown metadata belong to explicit task updates or UI editing flows.

Created open tasks with no due or scheduled time receive a `follow_up_at` stale-review timestamp seven days later.
That keeps an inbox-style task actionable immediately while giving the UI and task queries a concrete time to surface aging clutter.
