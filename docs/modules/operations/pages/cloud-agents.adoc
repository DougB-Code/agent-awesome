= Cloud Agents
:description: Step-by-step provisioning for per-agent Cloudflare deployments.

Cloud agents are provisioned by `agent-awesome-provision`.
Each agent gets an isolated Worker name, hostname, R2 bucket, snapshot key, and generated internal tokens.

== Build Or Select The Provisioner

Use the built binary:

[source,sh]
----
cd /home/doug/dev/agentawesome/agent
./build/package/bin/agent-awesome-provision --help
----

Or use `go run` from the module:

[source,sh]
----
cd /home/doug/dev/agentawesome/agent/provision
go run ./cmd/agent-awesome-provision
----

The top-level help exits non-zero when no command is provided.
That is expected.

== Initialize Platform Defaults

[source,sh]
----
cd /home/doug/dev/agentawesome/agent/provision
go run ./cmd/agent-awesome-provision platform init \
  --zone-name agent-awesome.com \
  --agent-hostname-suffix agent-awesome.com \
  --cloudflare-account-id <account-id> \
  --worker-source-dir /home/doug/dev/agentawesome/agent
----

Verify the saved non-secret config:

[source,sh]
----
go run ./cmd/agent-awesome-provision platform show
----

== Store External Credentials

[source,sh]
----
go run ./cmd/agent-awesome-provision credentials set CLOUDFLARE_API_TOKEN
go run ./cmd/agent-awesome-provision credentials set OPENAI_API_KEY
----

Only store Slack credentials when the agent will receive Slack events:

[source,sh]
----
go run ./cmd/agent-awesome-provision credentials set SLACK_SIGNING_SECRET
go run ./cmd/agent-awesome-provision credentials set SLACK_BOT_TOKEN
----

For a per-agent Slack pilot, the Slack Events API request URL should match the derived agent hostname:

[source,text]
----
https://sister.agent-awesome.com/slack/events
----

The provisioner generates `AGENTAWESOME_GATEWAY_TOKEN` and `AGENTAWESOME_PERSISTENCE_TOKEN` per agent.
Do not set those manually.

== Dry Run

[source,sh]
----
go run ./cmd/agent-awesome-provision cloudflare apply \
  --agent-id sister \
  --slack \
  --dry-run
----

The dry run writes generated deployment files under `build/provision/sister` and prints planned operations without calling Cloudflare or reading real secrets.

== Apply

Make sure Docker is running and Node 22 or newer is active before applying.

[source,sh]
----
go run ./cmd/agent-awesome-provision cloudflare apply \
  --agent-id sister \
  --slack
----

Expected derived resources for `sister`:

[cols="1,2",options="header"]
|===
|Resource |Value

|Worker
|`agent-awesome-sister`

|Hostname
|`sister.agent-awesome.com`

|R2 bucket
|`agent-awesome-sister-memory`

|Snapshot key
|`memory/memory/context-snapshot.tar.gz`
|===

On first deploy, Cloudflare Containers can take several minutes before health checks succeed.
The provisioner waits for gateway health until `--health-timeout` expires.
Increase it for first-time deploys:

[source,sh]
----
go run ./cmd/agent-awesome-provision cloudflare apply \
  --agent-id sister \
  --slack \
  --health-timeout 5m
----

== Verify

[source,sh]
----
go run ./cmd/agent-awesome-provision agent list
go run ./cmd/agent-awesome-provision agent status sister
----

For JSON automation:

[source,sh]
----
go run ./cmd/agent-awesome-provision agent status sister --json
----

== Delete

Preview deletion first:

[source,sh]
----
go run ./cmd/agent-awesome-provision agent delete sister --dry-run
----

Then delete remote Cloudflare resources and local state:

[source,sh]
----
go run ./cmd/agent-awesome-provision agent delete sister
----

The command requires exact typed confirmation.
Use `--force` only for non-interactive automation.

Use `--local-only` only when remote resources were already removed and the local record or generated tokens need cleanup.
