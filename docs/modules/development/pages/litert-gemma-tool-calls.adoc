= LiteRT Gemma Tool Calls
:description: How the LiteRT adapter keeps Gemma tool calls compatible with ADK and the local model context window.

This page documents the local Gemma tool-call fix so future changes do not reintroduce prompt-only tool handling or oversized LiteRT prompts.

== Runtime Contract

ADK owns chat orchestration and tool execution.
The LiteRT adapter's job is only to translate the ADK `LLMRequest` into Gemma's text protocol, run `litert-lm`, and translate valid Gemma tool-call text back into ADK `FunctionCall` parts.

Do not move tool execution, idempotency policy, MCP calls, or chat loop logic into the LiteRT prompt.
Those responsibilities belong to ADK, runtime callbacks, and tool implementations.

== Gemma Format

Gemma tool support uses chat-template control tokens, not a natural-language `AVAILABLE TOOLS` prompt.
Tool declarations are serialized in the system turn:

[source,text]
----
<|tool>declaration:create_task{parameters:{type:<|"|>OBJECT<|"|>}}<tool|>
----

Model tool calls are parsed from:

[source,text]
----
<|tool_call>call:create_task{title:<|"|>Buy milk<|"|>}<tool_call|>
----

The adapter then returns a real `genai.FunctionCall` to ADK.
ADK executes the tool and sends the resulting `FunctionResponse` back through the next model turn.

The upstream Gemma function-calling format is documented by Google at https://ai.google.dev/gemma/docs/capabilities/function-calling[].

== Prompt Size Fix

The bug was not a missing binary or broken model.
`litert-lm` was exiting with:

[source,text]
----
Input token ids are too long. Exceeding the maximum number of tokens allowed: 6353 >= 4096
----

The harness was serializing the full MCP catalog into every local-model prompt.
That exceeded the `gemma-4-E2B-it` context window before Gemma could produce a tool call.

The fix is in `harness/internal/model/adapters/litert`:

* `litert.go` builds the Gemma chat-template prompt and captures bounded `litert-lm` stderr on failures.
* `tool_selection.go` selects a compact set of relevant tool declarations for the current user turn and preserves declarations needed by active tool history.
* Tool-call parsing still emits ADK `FunctionCall` parts. It does not call MCP tools directly.

For example, "Make a reminder to buy milk" should select `create_task` instead of sending every memory, search, graph, relation, and local-exec schema to Gemma.

== Regression Checks

Run these checks after touching LiteRT prompt building, Gemma tool parsing, MCP tool declarations, or runtime callbacks:

[source,sh]
----
cd /home/doug/dev/agentawesome/agent/harness
go test -count=1 ./internal/model/adapters/litert
go test -count=1 ./...
----

For an end-to-end local check, start the UI or a debug harness with `gemma-4-E2B-it` and send:

[source,text]
----
Make a reminder to buy milk
----

Expected behavior:

* Gemma emits a `create_task` tool call.
* ADK executes the MCP tool.
* The final assistant message confirms the created task.
* No `Input token ids are too long` error appears.
