= Documentation Architecture
:description: How Agent Awesome uses Antora and distributed component roots.

Agent Awesome uses one Antora component named `agent-awesome`.
The component is distributed across several nearby content roots so documentation can live beside the code it describes while still publishing into one consolidated site.

== Content Roots

[cols="1,2",options="header"]
|===
|Path |Owns

|`docs`
|Site metadata, navigation, user docs, internal development docs, internal launchpad docs, and cross-system overview pages.

|`harness/docs`
|Harness module pages beside the harness Go module.

|`memory/docs`
|Memory module pages beside the memory Go module.

|`gateway/docs`
|Gateway module pages beside the gateway Go module.

|`ui/docs`
|UI module pages beside the Flutter app.
|===

Only `docs/antora.yml` owns optional component metadata such as `title`, `start_page`, and `nav`.
The other `antora.yml` files identify distributed content roots by component name and version only.

== Build

Build the documentation site from the repository root:

[source,sh]
----
npm install --cache ./build/npm-cache
npm run docs:build
----

The generated site is written to `build/site`.
Antora cache data is written to `build/antora-cache`.

== Writing Rules

Place code-specific documentation in the nearest subsystem docs root.
Place end-user runbooks in `docs/modules/user`.
Place internal development runbooks in `docs/modules/development`.
Place local support and repair procedures in `docs/modules/launchpad`.

Do not duplicate user, development, and launchpad instructions in one page.
When a page must mention both audiences, link to the other page instead of embedding both runbooks.
